## 缘起

过去几个月，我用 Claude Code 和 OpenCode 分别做了两个纯粹给自己用的项目：一个记账本，一个量化交易小应用。这两个项目本身不复杂，但做完之后，我对"软件"这件事的理解发生了一些微妙但重要的变化。

## 记账本：从"想要"到"可用"只花了一个晚上

我对市面上的记账 App 一直不满意。要么功能太臃肿，要么分类逻辑和我的习惯不一样，要么在隐私上让我不放心。以前这种"不满意"只能忍着，因为自己写一个的成本太高了。

用 Claude Code 之后，我第一次体验到了"用自然语言描述需求，然后看着代码一点点长出来"的感觉。我告诉它我想要什么样的分类体系、什么样的统计视图、什么样的数据存储方式，它就一步步把这些东西搭起来了。

过程当然不是完美的。Agent 有时候会过度设计——我只要一个简单的 SQLite 本地存储，它非要给我加一层 ORM；我只想要一个朴素的命令行界面，它老想给我搭一个 Web UI。但这些都是可以纠正的。更重要的是，**纠正的成本极低**——我用自然语言说一句"不要这个，简单点"，它就改了。

最终的记账本完全按我的心智模型来组织：分类是我习惯的分类，统计是我关心的统计，数据就在我自己的机器上。这种"完全贴合"的感觉，是任何通用软件都给不了的。

## 量化应用：Agent 的上限和工程师的价值

量化交易应用更有挑战性。它涉及到数据获取、策略回测、风险管理、实盘对接，每一块都有自己的坑。

这个项目让我看到了 Agent 的能力边界在哪里。**在"骨架搭建"这个层面，Agent 非常高效。** 数据管道、回测框架、基本的策略模板，这些结构化程度高的工作，Agent 做得又快又好。

但到了策略逻辑的细节，Agent 的价值就变成了"一个非常好的讨论伙伴"而不是"一个能替你写策略的工具"。因为策略本身是对市场的理解，这种理解没法被简单地描述成 prompt。我发现自己最高效的工作模式是：用 Agent 快速验证一个想法——"帮我写一个基于 XXX 的均值回归策略，先跑一下近三年的回测"——然后根据结果来调整方向。

OpenCode 和 Claude Code 的风格差异也很有意思。Claude Code 更"主动"，会预判你的需求并提前给出建议；OpenCode 更"忠实"，严格按你的要求来。两者我都在用，根据场景切换。

## 软件的形态会根本性地改变吗？

做完这两个项目之后，我开始认真思考一个问题：**如果每个人都能用 AI 轻松地构建完全贴合自己需求的软件，那"软件"这个东西的形态会怎么变？**

### 技术视角：从标准化到个性化

软件工程的历史，某种程度上是一部标准化的历史。我们做通用软件、做 SaaS、做平台，核心逻辑都是"用标准化的方案服务最多的人"。这是工程效率的必然选择——当软件开发成本高昂时，标准化是唯一经济可行的路径。

但 AI Agent 正在把软件开发的边际成本推向接近零。当开发成本不再是瓶颈时，"为什么要用一个通用方案来将就？"就变成了一个合理的问题。

我能想象一种未来：**很多现在由 App Store 里的应用满足的需求，会被用户自己的 Agent 生成的个性化工具所替代。** 不需要下载，不需要注册，不需要适应别人的设计——Agent 直接根据你的需求和习惯，生成一个只属于你的工具。

当然，这不意味着所有软件都会消失。复杂的基础设施（数据库、操作系统、云服务）、需要强网络效应的平台（社交、电商）、安全要求极高的系统（金融、医疗），这些仍然需要专业团队来构建和维护。但"中间层"——那些工具类、效率类、个人管理类的应用——会经历一次大规模的去中心化。

### 哲学视角：工具的"合手性"

海德格尔有个概念叫"上手"（Zuhandenheit），说的是当一个工具完全贴合你的使用习惯时，你会"忘记"它的存在，它成为你身体的延伸。一把好用的锤子在木匠手里是"透明的"——木匠不会去想锤子本身，他想的是钉子和木头。

现在大多数软件都不是"上手"的。我们花大量时间在"学习使用"而不是"使用"上。每换一个 App 都要重新适应。这不是因为设计师无能，而是因为通用软件的本质矛盾：它必须服务所有人，所以它不可能完美贴合任何一个人。

AI 生成的个性化工具，第一次有可能让软件真正变得"上手"。因为它就是按你的思维方式来的，不存在适应过程。我在用自己的记账本时就有这种感觉——它不像是一个"软件"，更像是一个自然的延伸。

这是一个深刻的变化。**当工具不再是你需要适应的外在物，而是可以完全贴合你的延伸时，人和技术的关系就变了。**

### 社会视角：新的不平等？

但我也必须诚实地说出另一面。

能够用 AI Agent 构建个性化工具的前提是：你知道自己想要什么，你能清晰地描述自己的需求，你有能力判断 Agent 的输出是否正确。这三项能力并不是均匀分布的。

一个有工程背景的人用 Claude Code 和一个完全没有技术背景的人用同样的工具，得到的结果天差地别。不是因为工具有门槛，而是因为"知道自己想要什么"和"能判断产出质量"本身就是一种能力。

这可能会形成新的数字鸿沟：不是"能不能用电脑"的鸿沟，而是"能不能有效地指挥 AI"的鸿沟。有人会获得高度定制的、完美贴合自己的工具集；有人则继续使用通用的、将就的方案。

教育需要跟上。**未来最重要的素养可能不是编程，而是"需求表达"——能够把模糊的感觉转化为清晰的描述，能够在 Agent 的多个输出中判断哪个更好，能够在迭代中逼近自己真正想要的东西。**

## 回到个人

写到最后，我想回到一些更个人的感受。

用 AI Agent 做项目的体验，让我想起刚学编程时的那种兴奋感——"原来我可以让机器做这个事情"。那种把想法变成现实的即时反馈，那种不断迭代逼近理想状态的过程，是纯粹的创造乐趣。

区别在于，现在从想法到现实的路径短了很多。以前可能需要几周的事情，现在可能一个晚上就能做出一个 MVP。这种压缩不只是效率的提升，它从根本上改变了你思考问题的方式——你会更愿意去尝试，更愿意去做那些"够不够得上做成一个正式项目"的小东西。

也许这才是 AI Agent 对个人最大的价值：**它降低了创造的门槛，让更多的想法有机会被实现，让"自己动手"从一种需要特殊技能的行为变成一种日常的、自然的行为。**

软件的未来也许不是更多的 App，而是更多的人在为自己构建刚好够用的工具。这听起来不像是一场革命，但它可能是。
