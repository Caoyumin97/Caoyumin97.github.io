## 一个反直觉的趋势

如果你最近在关注开发工具的演化，你会注意到一个看起来"倒退"的趋势：**一切都在 CLI 化。**

Claude Code 是一个终端应用。OpenCode 是一个终端应用。Cursor 虽然穿着 IDE 的外衣，但核心交互是在一个文本框里打字。就连很多新的 AI 产品，底层都在走 MCP（Model Context Protocol）——本质上就是让 Agent 可以调用的一组命令行接口。

这不是审美复古。这是因为软件的"用户"正在发生根本性的变化。

## 从 Human-Centric 到 Agent-Centric

过去三十年的软件设计，核心假设是：**用户是人。** 所以我们需要图形界面、需要动效、需要直觉化的交互、需要 onboarding 流程。一个好的产品经理会花大量时间在用户旅程地图上，考虑每一个点击路径上的认知负荷。

这些在人类用户的场景下都是对的。但当用户变成 AI Agent 时，这些东西不仅没用，还是障碍。

Agent 不需要好看的按钮。Agent 不需要引导页。Agent 不需要动效来缓解等待焦虑。Agent 需要的是：**清晰的输入输出契约、可预测的行为、原子化的能力单元、以及机器可读的描述。**

换句话说，Agent 需要的就是一个设计良好的 CLI——或者更准确地说，一个设计良好的 API。

## "CLI 化"的本质是接口的去装饰化

我说的 CLI 化不是字面意义上的"所有东西都变成命令行工具"。它的本质是：**软件正在把自己的能力以最小装饰、最大机器可读性的方式暴露出来。**

想想 MCP 协议在做什么。它定义了一种标准方式，让任何服务都可以把自己的能力描述成一组"工具"——每个工具有名字、有参数 schema、有返回值类型。Agent 读到这些描述，就知道怎么用了。

这不就是 Unix 哲学的 AI 时代翻版吗？

- 每个程序做好一件事（原子能力）
- 用文本流作为通用接口（结构化的输入输出）
- 程序之间可以自由组合（Agent 编排）

区别在于，Unix 时代的"组合者"是写 shell 脚本的人类工程师，而现在的"组合者"是 AI Agent。Agent 比人类更擅长处理大量工具的组合——它不会忘记参数，不会搞错管道，不会因为工具太多而认知过载。

但这也意味着，**Agent 对工具的质量要求比人类更苛刻。** 人类可以看着一个设计不太合理的 GUI 凑合着用，因为人有常识可以补全。Agent 遇到一个描述模糊的 API 就会直接出错或者幻觉。

## 原子能力：下一个时代的核心基建

如果 Agent 是新的"组合者"，那我们应该把精力花在哪里？

答案越来越清晰：**构建高质量的原子能力。**

什么是原子能力？它是一个功能单元，满足几个特征：

1. **单一职责。** 它只做一件事，但把这件事做到极致。不是"帮你管理项目"，而是"创建一个 issue"、"查询一个 issue 的状态"、"修改一个 issue 的标签"。
2. **清晰的边界。** 输入是什么、输出是什么、副作用是什么，都有明确的定义。没有"看情况"、没有"通常来说"。
3. **可组合性。** 它的输出可以直接成为另一个原子能力的输入，不需要人工转换或解释。
4. **幂等或可预测。** 同样的输入永远产生同样的输出（或至少行为是可预测的）。Agent 需要信任它调用的工具。

在供应链场景里，我们正在做的事情就是把一整套复杂的业务系统拆解成原子能力。以前我们想的是"做一个好用的库存管理系统"，现在我们想的是"把库存管理的每一个原子操作都做成 Agent 可调用的工具"。

查询某个 SKU 在某个仓库的实时库存——这是一个原子能力。
根据安全库存水位生成补货建议——这是一个原子能力。
向供应商发送一个采购订单——这是一个原子能力。

Agent 来决定什么时候调用哪个、以什么顺序组合、中间要不要加入判断逻辑。**我们不再需要预先设计"流程"，我们只需要提供可靠的"积木"。**

## 横向能力 vs 纵向场景

这里有一个有趣的分工问题。

**纵向场景**（行业场景、业务逻辑）的 know-how 仍然掌握在人类手中。Agent 不知道什么时候该催供应商、什么时候该换物流商，这些决策依赖于行业经验和情境判断。

但**横向能力**（数据查询、文件处理、消息通知、审批流转、报表生成……）正在快速地标准化和原子化。这些是每个业务场景都需要的基础能力，它们的共性远大于差异。

我的判断是：**横向原子能力的构建，是当前最值得投入的基础设施工作。** 因为：

- 一旦建好，所有纵向场景都能复用。
- Agent 在横向能力的组合上已经非常强了，不需要太多领域知识。
- 这些能力的质量直接决定了 Agent 在上层应用中的表现上限。

这有点像云计算早期。当年大家在讨论"SaaS 还是 PaaS 还是 IaaS"时，真正的赢家是那些把基础能力（计算、存储、网络）做到极致可靠、极致标准化的人。在 Agent 时代，赢家可能是那些把原子能力做到极致可靠、极致可描述的人。

## 产品经理要重新想自己的工作了

这种转变对产品设计的影响是深远的。

传统的产品设计思维是：理解用户需求 → 设计信息架构 → 画交互流程 → 打磨视觉细节。每一步都围绕"人类用户如何感知和操作"展开。

Agent-Centric 的产品设计思维可能变成：**定义原子能力 → 设计能力间的组合规则 → 编写机器可读的能力描述 → 确保每个能力的可靠性和可预测性。**

这不是说 GUI 会消失。人类仍然需要界面来监督 Agent、做高层级的决策、处理异常情况。但 GUI 可能从"主界面"变成"监控台"——你不再通过它来完成日常操作，你通过它来观察 Agent 的运行状态、在需要的时候介入。

产品经理的核心能力也在变：不再是"能画出好看的原型"，而是"能定义出清晰的能力边界"。不再是"理解用户点击路径"，而是"理解 Agent 的调用链路"。

## 一个类比：从手工作坊到工业流水线

如果要用一个历史类比的话，当前的转变有点像从手工作坊到工业流水线的过渡。

手工作坊时代，工匠掌握完整的技能链——从原料到成品。工具是为工匠的手设计的，每一件都有人体工学的考量。

工业时代，生产被拆解成标准化的工序。每个工序是一个原子操作，由机器执行。人类的角色从"执行者"变成"设计者"和"监督者"。工具不再需要握感好，它需要接口标准、输出一致。

我们现在经历的是软件行业的同一个过程。Agent 就是新的"机器"，原子能力就是"标准化工序"，而我们——工程师和产品设计者——正在从"手工写每一行代码、设计每一个交互"的工匠，变成"定义能力、设计组合规则、监督 Agent 执行"的工程架构师。

## 回到第一性原理

说到底，这一切的底层逻辑很简单：

**当消费者（用户）从人变成了 Agent，生产者（开发者）的工作重心自然要从"人机交互"转向"机机交互"。**

而机机交互的核心就三件事：
1. 把能力做原子化（每个积木足够小、足够可靠）
2. 把描述做标准化（让 Agent 能准确理解每个积木能干什么）
3. 把组合做自由化（不预设流程，让 Agent 根据目标自行编排）

CLI 化、MCP、原子能力——这些不是独立的趋势，它们是同一个逻辑的不同表达。

作为从业者，我觉得现在最有价值的事情是：**停止执着于为人类用户打磨界面，开始为 Agent 构建可靠的能力积木。** 这不意味着放弃用户体验——而是认识到，未来的用户体验可能更多地由 Agent 的编排质量决定，而编排质量又由底层原子能力的质量决定。

把地基打好，Agent 会处理剩下的事情。
